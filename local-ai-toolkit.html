A field guide for local-AI tinkerers

Run the whole frontier on one quiet machine.

Everything ChatGPT and Claude do for you — code execution, file creation, search, research, memory, voice — is mostly scaffolding around the model. This is the catalogue of that scaffolding, and the open-source piece you can self-host in its place.

For a Qwen-class model on an Apple M1 Max · 32 GB unified memory · 1 TB SSD

TL;DR Key findings The 36 capabilities The host layer Staged plan Caveats

The short version

Nearly every frontier "product" capability can be rebuilt locally with open-source MCP servers and Python libraries. The pieces you hadn't catalogued are mostly the agentic ones — computer use, scheduled tasks, deep-research agents, sub-agent orchestration, "Skills" as a packaging format, voice mode, and the citation/structured-output plumbing — and almost all have a credible self-hostable equivalent.

The real ceiling isn't tooling, it's your 32 GB of unified memory. A Qwen dense 14B or a 30B-A3B MoE at Q4 (≈9–19 GB) leaves room for tools — but image generation, a local vision model, and a voice pipeline all share that memory with your LLM. Run them in sequence, not all at once, and treat heavy image/video work as a "swap the model out" task.

The stack that works: serve Qwen through llama.cpp / Ollama / LM Studio (needed for reliable tool-calling — MLX is faster but can't call tools yet), front it with LibreChat or Open WebUI (both speak MCP natively and give you artifacts, RAG and voice), then bolt on community MCP servers for search, fetch, code, files, PDF, memory and RAG.

Key findings

The frontier products are agent platforms now, not chatbots.

As of mid-2026, ChatGPT and Claude have both moved well past the categories you already had. The capabilities you hadn't explicitly listed — and which matter most — are these:

  1. Computer use & agentic browser control — Claude Computer Use, Claude for Chrome, Claude Cowork (autonomous access to a local folder), and OpenAI's ChatGPT Agent (formerly Operator) driving a VM with browser, terminal and APIs.
  2. Deep Research — both spend 5–30 minutes planning sub-questions, searching, reading and synthesising a cited multi-page report.
  3. Scheduled tasks / automations — ChatGPT "Tasks" and Cowork recurring jobs (cron-like).
  4. Anthropic "Skills" — folders with a SKILL.md (YAML frontmatter + Markdown) plus optional scripts and assets, loaded progressively so the frontmatter costs only ~100 tokens. The docx/pdf/pptx/xlsx skills that power Claude's file creation are published, and the format became an open standard in December 2025.
  5. Sub-agent orchestration — Cowork coordinates parallel sub-agents; OpenAI AgentKit is a visual multi-agent builder.
  6. Voice / realtime, inline citations, structured outputs & function calling, MCP connectors (Gmail, Drive, GitHub, Slack…), and chart/diagram generation round out the list.

Your model choice is fine — run it on the right engine

The open-weight lineage as of June 2026 is Qwen3 (Apr 2025), the Qwen3-2507 refresh, Qwen3.5 (Feb 2026, native-multimodal), and Qwen3.6 (Apr 2026). The "Max / Plus" tiers are API-only — not for local use. Here's what actually fits 32 GB at Q4 with headroom left for the OS and tools:

ModelQ4 footprintVerdict
Qwen3-14B (dense)≈ 9.3 GBsafe · most headroom
Qwen3-30B-A3B (MoE)≈ 17–19 GBsweet spot
Qwen3.6-27B (dense vision)≈ 16.8 GBsafe
Qwen3-32B (dense)≈ 20 GBtight · short context
Qwen3.5 / 3.6-35B-A3B≈ 21 GBtight
Qwen3-235B-A22B≈ 142 GBwon't fit

The tool-calling caveat is the one to remember. Use llama.cpp (with --jinja), Ollama, or LM Studio for tool-calling agents — MLX is faster on Apple Silicon but doesn't support tool calling yet. Qwen's official chat templates have recurring tool-calling bugs, so prefer Unsloth's quants (they ship fixed templates) or community-fixed templates, and keep llama.cpp updated. The 30B-A3B MoE only activates ~3B parameters per token, which makes it genuinely fast on your hardware — it's the one to start with.

The catalogue

36 capabilities, and the open-source piece that replaces each.

Each card maps one frontier capability to its self-hostable equivalent, with a note on whether it's realistic on your 32 GB M1 Max. The numbering is the index, not a ranking — work through whichever matters to you first. Expand the "details" toggle on any card for how the pieces actually fit together, where the tool earns its keep, and a prompt to try on day one.

01

Code interpreter & data analysis

Frontier

ChatGPT's sandboxed Python (Advanced Data Analysis) and Claude's Ubuntu code sandbox — run Python/Node, make charts, accept CSV/Excel/PDF.

Open source

Open Interpreter for natural-language local execution against Ollama; llm-sandbox or microsandbox for Docker/microVM isolation; or just a constrained Docker container running a Jupyter kernel.

On your M1 Max

Trivial — Docker runs ARM containers fine, and CPU-bound Python is no problem.

details & example prompt How it works

Your chat UI hands the model a "run code" tool. The model writes Python, the sandbox executes it, and stdout / errors / saved plots flow back as tool results — so the model can see a traceback and fix its own code over several rounds. Files you upload land in the sandbox's working directory.

Where it's most useful

CSV/Excel analysis, quick plots, batch file conversion and renaming, regex-heavy text munging — anything where "just write the script" beats explaining.

Try this prompt

Here's sales.csv — load it, drop rows with missing totals, plot monthly revenue as a bar chart, and tell me the three strongest months and one trend I might have missed.

02

File generation (docx / xlsx / pptx / pdf)

Frontier

Claude's file creation is literally code execution plus the published office Skills; ChatGPT Canvas exports PDF/MD/DOCX.

Open source

Run inside your sandbox: python-docx, openpyxl, python-pptx, reportlab / WeasyPrint (HTML→PDF), and pandoc. Clone anthropics/skills to reuse Claude's exact workflows.

On your M1 Max

All pure-Python — negligible memory.

details & example prompt How it works

There is no magic "make docx" model — the LLM writes python-docx / openpyxl / python-pptx code in the sandbox, runs it, and the UI offers the saved file for download. The Anthropic office Skills are essentially battle-tested recipes for exactly this, which is why cloning them saves you weeks of formatting bugs.

Where it's most useful

Weekly reports, invoices, formatted proposals, spreadsheets built from pasted data, slide decks from an outline.

Try this prompt

Create a 5-slide pptx pitch for a neighborhood bakery: title, problem, solution, a pricing table, and contact. Clean minimal styling, one accent color.

03

PDF & document parsing

Frontier

Native PDF ingestion via combined vision + text.

Open source

Docling (structured + OCR, best for RAG), PyMuPDF4LLM (fastest on digital PDFs), Marker (deep-learning, Apple MPS support), MinerU (CJK/scientific), pdfplumber (table coordinates).

On your M1 Max

Marker uses MPS; Docling/PyMuPDF run comfortably on CPU. Heavy OCR adds load — fine sequentially.

details & example prompt How it works

The parser converts a PDF into structured Markdown/JSON — headings, tables, reading order — falling back to OCR for scanned pages. That clean text is what actually goes into the model's context (or your RAG index); the model never "sees" the raw PDF bytes.

Where it's most useful

Contracts, research papers, scanned invoices and statements, and as the ingestion step for every knowledge base you build (card 14).

Try this prompt

Parse this 40-page PDF, extract every table into one xlsx (one sheet per table), and summarize section 3 in plain English with the key numbers.

04

Web search

Frontier

Built-in search with inline citations.

Open source

SearXNG — self-hosted metasearch, Docker, no API key, the privacy-first default. Cheap cloud options: Tavily, Brave Search API.

MCP server

mcp-searxng is the well-regarded bridge and supports failover replicas.

On your M1 Max

A lightweight container — no GPU, minimal RAM.

details & example prompt How it works

The model calls a search tool; the MCP server forwards the query to your SearXNG container, which fans it out to Google/Bing/DuckDuckGo and returns deduplicated titles, snippets and URLs. The model reads the snippets, decides which pages are worth a full fetch (card 5), and answers with links.

Where it's most useful

Anything after your model's training cutoff — current versions, prices, news, error messages — plus fact-checking the model's own claims.

Try this prompt

Search for the currently recommended llama.cpp flags for Qwen3 MoE models, compare the top three sources, and give me the consensus command with links.

05

Web fetch / browsing / scraping

Frontier

Web fetch plus full agentic browsing (Operator / Computer Use).

Open source

Playwright (the standard), readability extractors, Firecrawl (self-hostable, Markdown output) for JS-heavy pages.

MCP server

Microsoft Playwright MCP — accessibility-tree based, so no vision model needed (~200–400 tokens per snapshot). Doubles as your fetch tool.

On your M1 Max

Headless Chromium keeps memory down; fine alongside a small LLM.

details & example prompt How it works

Playwright MCP drives a real (headless) Chromium and hands the model a compact accessibility-tree snapshot of the page instead of a screenshot — a few hundred tokens listing links, buttons and text. The model picks an action (click, type, navigate), gets a fresh snapshot, and loops. JS-rendered content just works because it's a real browser.

Where it's most useful

Reading JS-heavy pages that plain HTTP fetch can't, pulling clean article text, checking a dashboard, light multi-step navigation.

Try this prompt

Open Hacker News, find today's top five stories about local LLMs, and give me a one-line take on each with the link.

06

Image generation

Frontier

DALL·E 3 / GPT-image in ChatGPT; Claude has none (uses code/SVG).

Open source

Draw Things (native Metal, ~20% faster on Apple Silicon) or ComfyUI running SDXL or Flux.1 (schnell for speed). Use GGUF-quantized models to fit memory.

On your M1 Max — the pinch point

Flux dev needs ~12 GB+; running it and a 14–30B LLM at once will blow past 32 GB. Unload the LLM first, or treat image-gen as a separate session. SDXL is more forgiving.

details & example prompt How it works

A text encoder turns your prompt into an embedding that steers a diffusion model as it denoises random latents into an image over 4–30 steps. ComfyUI exposes that pipeline as a node graph (model → sampler → VAE decode), so you can swap checkpoints, add LoRAs or control poses; Draw Things wraps the same idea in a native Mac app.

Where it's most useful

Blog headers, thumbnails, moodboards, concept art, product mockups — anywhere "good enough, private and free per-image" beats a cloud subscription.

Try this prompt

A cozy isometric illustration of a small home-server rack glowing at night, soft pastel palette, subtle film grain, 1024×1024.

07

Vision / image understanding

Frontier

Native vision — charts, screenshots, handwriting, PDFs.

Open source

Qwen3-VL (4B/8B/30B-A3B; OCR in 32 languages, video, GUI-agent skills), plus LLaVA, Pixtral, Gemma vision. Run via mlx-vlm (Apple-native) or llama.cpp.

On your M1 Max

VL-4B/8B at 4-bit fit comfortably; 30B-A3B is tight at Q4. For vision and tool-calling, prefer llama.cpp/LM Studio over MLX.

details & example prompt How it works

A vision encoder slices the image into patches and projects them into the same token space the LLM reads — so the model literally attends over your screenshot the way it attends over words. Higher-resolution modes cost more tokens but read small text (UI labels, receipts) far better.

Where it's most useful

Debugging from screenshots, reading charts and dashboards, receipts and forms, handwriting, "what is this thing?" photos.

Try this prompt

Here's a screenshot of my terminal — what's actually failing, and what exact command should I run next?

Canvas / Artifacts

Frontier

Live HTML / React / SVG / Mermaid preview, embeddable.

Open source

Open WebUI, LibreChat and LobeChat all render artifacts inline with persistent storage.

On your M1 Max

Pure front-end — no extra model load.

details & example prompt How it works

The UI watches the model's output for renderable blocks — HTML, React, SVG, Mermaid — and mounts them in a sandboxed iframe beside the chat. Ask for a change and the model rewrites the code; the preview re-renders live. The artifact persists with the conversation, so you can come back and iterate.

Where it's most useful

Prototyping small web tools, one-file games, SVG logos, architecture diagrams, interactive explainers you can hand to someone as a single file.

Try this prompt

Build a single-file HTML pomodoro timer with a circular progress ring, keyboard shortcuts, and a subtle tick sound. Dark theme.

09

Persistent memory

Frontier

Cross-chat and per-Project memory in both products.

Open source

mem0 (light bolt-on; extracts facts to a vector DB) or Letta / MemGPT (full stateful runtime with OS-style tiered memory and notably higher recall accuracy, at more overhead). Zep/Graphiti for temporal graphs.

MCP server

Official server-memory (knowledge-graph), plus mem0 / Qdrant-backed servers.

On your M1 Max

Light — the cost is one extra LLM call per write for fact extraction.

details & example prompt How it works

After an exchange, a small extraction call pulls out durable facts ("prefers metric", "server is called atlas"), deduplicates them against what's stored, and writes them to a vector DB or knowledge graph. At the start of each new chat, the memories most relevant to your message are retrieved and injected into the system prompt — the model doesn't "remember", it re-reads.

Where it's most useful

Preferences and house style, ongoing projects, names and context of people you mention often — anything you're tired of re-explaining.

Try this prompt

Remember: I prefer metric units, my home server is called "atlas", and I want answers to default to bullet points. Apply this from now on.

Custom instructions & personalization

Frontier

Custom instructions, Custom GPTs, Projects with shared setup.

Open source

System prompts and per-conversation presets in LibreChat / Open WebUI; their "Agents / Projects" wrap a model with instructions + tools + knowledge. SKILL.md files double as reusable personalization.

details & example prompt How it works

A preset is just stored text that gets prepended as the system prompt every time — but bundled with a model choice, tool list and attached knowledge files, it behaves like a Custom GPT. You pick the preset from a dropdown instead of re-pasting instructions.

Where it's most useful

One agent per recurring role: code reviewer, email drafter, meal planner, "explain like I'm five" tutor — each with its own tone and tools.

Try this prompt

Create an agent preset called "Reviewer": a terse senior Python reviewer who always checks error handling, suggests type hints, and ends with a one-line verdict.

Connectors (Gmail, Drive, GitHub…)

Frontier

Native connectors, now delivered over remote MCP.

Open source

Official reference servers (filesystem, git, fetch, memory, time) plus community servers for GitHub, Google Drive/Calendar, Slack, Notion, Supabase and more. Browse the official MCP Registry, PulseMCP, Glama, Smithery.

On your M1 Max

I/O-bound API wrappers — negligible local use.

details & example prompt How it works

Each MCP server wraps one service's API and advertises its actions as tools ("list issues", "search drive", "send message"). Your chat UI lists those tools to the model; when the model calls one, the server executes it with credentials you configured and returns the result. Auth stays on your machine — the model only ever sees tool names and outputs.

Where it's most useful

Triaging GitHub issues and PRs, searching your Drive/Notion, posting summaries to Slack, calendar lookups — the "do it where my stuff lives" layer.

Try this prompt

Check my GitHub notifications, group them by repo, and draft a short reply for anything that mentions me directly.

Computer use / browser agents

Frontier

Claude Computer Use / Cowork / Chrome; ChatGPT Agent.

Open source

browser-use (leading OSS browser agent), Playwright MCP for vision-free browsing, Stagehand, and self-hostable OpenClaw.

On your M1 Max

Works, but small models are weaker at long-horizon GUI tasks — expect more retries. Qwen3-VL's GUI skills help. Headline OSS benchmarks use a frontier brain; a local Qwen scores lower.

details & example prompt How it works

An observe-act loop: the agent takes a snapshot of the page (accessibility tree, or a screenshot for vision models), the model chooses one action — click, type, scroll, navigate — the browser executes it, and a fresh snapshot comes back. Repeat until the goal is met or a step budget runs out. Guardrails like allowed-domain lists keep it from wandering.

Where it's most useful

Form filling, repetitive web chores (renewals, downloads, status checks), smoke-testing your own web apps.

Try this prompt

Go to my library's website, renew every book due this week, and tell me the new due dates. Stop and ask me before anything that costs money.

13

Deep research agents

Frontier

ChatGPT / Claude Deep Research.

Open source

GPT Researcher (cited multi-page reports, MCP server, works with local Ollama), LangChain Open Deep Research, and Local Deep Research (fully local with Ollama + SearXNG).

On your M1 Max

Runs locally; quality scales with model size. A 14–30B Qwen is decent but slower and less reliable than frontier research.

details & example prompt How it works

A planner call breaks your question into sub-questions; each one triggers search (card 4) and fetch (card 5), and the findings are written to a running set of notes. The loop checks coverage — "what's still unanswered?" — and searches again until satisfied, then a final writer pass turns the notes into a structured, cited report. Expect 10–40 minutes locally.

Where it's most useful

Buying decisions, technology comparisons, "state of X in 2026" surveys, literature scans — questions too broad for one search.

Try this prompt

Research e-ink tablets under $400: compare note-taking latency, ecosystem lock-in and battery life, cite every claim, and end with one recommendation for a heavy note-taker.

14

Projects / knowledge bases / RAG

Frontier

File collections with retrieval; "Company Knowledge."

Open source

LanceDB or Chroma (lightest, embedded) and Qdrant (production, hybrid search, official MCP server). Pair with FastEmbed; LibreChat / Open WebUI have built-in RAG.

On your M1 Max

LanceDB / Chroma embedded are the lightest; embeddings run on CPU/MPS.

details & example prompt How it works

Ingestion chunks each document (~300–800 tokens), embeds every chunk into a vector, and stores them. At question time your query is embedded too, the top-k nearest chunks are pulled (often combined with keyword search — "hybrid"), and those chunks are pasted into the model's context with instructions to answer from them and cite. Good chunking + reranking matters more than the vector DB brand.

Where it's most useful

Product manuals, your own notes and wikis, codebases, past invoices — private corpora too big for one context window.

Try this prompt

Using my "home-lab" knowledge base: how did I configure the reverse proxy last time, which ports did I map, and what did I say I'd do differently?

15

Voice (STT / TTS / realtime)

Frontier

Advanced Voice Mode — realtime speech-to-speech.

Open source

whisper.cpp (STT, full Metal acceleration) + Kokoro or Piper (TTS). MCP servers: local-stt-mcp, VoiceMode.

On your M1 Max

whisper.cpp uses Metal; TTS runs on CPU. Great for turn-based voice; true realtime alongside a big LLM is harder.

details & example prompt How it works

A three-stage pipeline: whisper.cpp streams your mic audio to text, the LLM generates a reply, and Kokoro/Piper synthesizes speech — with the TTS starting on the first sentence while the LLM is still writing the rest, which is what makes it feel responsive. Open WebUI has this wired in; VoiceMode does it over MCP.

Where it's most useful

Hands-free capture (cooking, driving, workshop), dictating notes and drafts, language pronunciation practice, accessibility.

Try this prompt

(spoken) Take a voice memo: three ideas for Saturday's newsletter. Then read them back to me as a numbered checklist.

Video understanding & generation

Frontier

Sora 2 (generation); Qwen3-VL / Gemini (understanding).

Open source

Understanding via Qwen3-VL frame sampling. Generation (HunyuanVideo, Wan 2.2 in ComfyUI) is possible but heavy.

On your M1 Max

Local video generation needs 64 GB+ — effectively out of scope here; treat as a separate-machine task.

details & example prompt How it works

For understanding, frames are sampled every 1–2 seconds and fed to the vision model as an image sequence while whisper transcribes the audio track; the model reasons over both. That's why long videos work at all — it's really "many screenshots plus a transcript", not a video codec in the model.

Where it's most useful

Summarizing lectures and recorded meetings, finding the moment something happens in a screen recording, cataloguing clips.

Try this prompt

Here's a 20-minute screen recording of me setting up the NAS — list each step I performed with timestamps, and point out where I wasted the most time.

Scheduled tasks & automations

Frontier

ChatGPT "Tasks"; Cowork recurring jobs.

Open source

n8n (fair-code workflows, 400+ integrations, can call your local LLM + MCP), or plain cron / launchd on macOS.

On your M1 Max

Light — orchestrates "make this report every morning" flows against Ollama.

details & example prompt How it works

A trigger (cron expression, webhook, file change) fires a workflow that calls your local model with a fixed prompt plus fresh inputs — search results, RSS items, yesterday's files — and delivers the output somewhere useful: a folder, an email, a Telegram/Slack message. n8n gives you this as drag-and-drop nodes; launchd + a shell script is the zero-dependency version.

18

Function calling & structured outputs

Frontier

Tool calling and JSON-schema structured outputs.

Open source

Native to your engine — llama.cpp, Ollama, vLLM, LM Studio support response_format and tool-calling (mind the Qwen template caveats). Frameworks: Qwen-Agent, LangChain, LlamaIndex.

details & example prompt How it works

Tool definitions (name, description, JSON schema) are formatted into the prompt by the chat template; the model emits a structured call, your code executes it and feeds the result back. For structured outputs, llama.cpp goes further: it constrains sampling with a grammar built from your schema, so invalid JSON is literally impossible to generate — the single most underrated local feature.

Where it's most useful

Every agent in this guide runs on this; also bulk data extraction — turning messy text into rows you can load into a spreadsheet or database.

Try this prompt

Extract vendor, date, currency, total and line items from this receipt text. Return only JSON matching: {vendor, date, currency, total, items:[{name, qty, price}]}.

19

Citations & source attribution

Frontier

Inline URL citations on search and research answers.

Open source

GPT Researcher and Local Deep Research produce citation-rich reports; your RAG layer returns source chunks to cite; SearXNG / Playwright supply the URLs.

details & example prompt How it works

Citations are plumbing, not intelligence: every search result and RAG chunk carries its URL or document ID through the pipeline, and the model is instructed to tag claims with [n] markers pointing back at them. The UI turns markers into links. If the source IDs aren't threaded through, the model will happily invent plausible-looking references — so this is about your pipeline, not your prompt.

Where it's most useful

Anything you'll act on or share — research reports, medical/financial lookups, "prove it" answers from your own document base.

Try this prompt

What changed in the latest Ollama release? Answer with inline [n] citations after each claim and a numbered source list with URLs at the end.

Charts, diagrams, visualization

Frontier

matplotlib via code interpreter; Mermaid / SVG / React via artifacts.

Open source

matplotlib / plotly in your sandbox; Mermaid & SVG rendered in Open WebUI / LibreChat artifacts; a Vega-Lite MCP server for data-driven visuals.

details & example prompt How it works

Two routes: for data, the model writes matplotlib/plotly code that runs in the sandbox (card 1) and returns a PNG or interactive HTML; for structure, it writes Mermaid or SVG markup that the artifact pane (card 8) renders directly — no execution needed. Translation, OCR & utilities

Translation, OCR and small utility tools built in.

Open source

Qwen handles translation; OCR via Tesseract / RapidOCR / EasyOCR (bundled in Docling/Marker) or Qwen3-VL's 32-language OCR. The official time, filesystem, git servers and pandoc cover the rest.

OCR engines return raw text plus layout boxes; the LLM then does what OCR alone can't — fixes recognition errors from context, restores paragraph structure, and translates while keeping formatting. For clean photos, a vision model (card 7) can do the whole read-and-translate step in one pass.

Foreign-language letters and forms, menu and sign photos, digitizing old scanned documents, quick format conversions via pandoc.

Coding agents & IDE assistants

Frontier

Claude Code, OpenAI Codex CLI, GitHub Copilot — agents that read a repo, edit files, run tests and open PRs.

Open source

Aider (git-native pair programmer, works with Ollama), OpenHands (full autonomous dev agent), Cline / Roo Code (VS Code agents), Goose (Block's MCP-native agent), Continue.dev + Tabby for local autocomplete.

On your M1 Max

Aider + Qwen3-30B-A3B is genuinely usable for small edits; a dedicated coder quant (Qwen3-Coder class) helps. Long multi-file refactors are where local models need the most supervision.

details & example prompt How it works

The agent builds a "repo map" (files, signatures, imports) so the model can orient without reading everything, then proposes edits as diffs, applies them, and runs your tests or linter. Failures go back into context and it retries. Aider commits each change to git automatically — your undo button is git revert.

Where it's most useful

Small features and bug fixes, writing tests, mechanical refactors, and "explain how X works in this codebase" on projects you didn't write.

Try this prompt

Add a --json flag to cli.py that switches all output to JSON, update the argument parser and README, and extend the existing CLI tests to cover it.

Sub-agent orchestration / multi-agent

Frontier

Claude Cowork's parallel sub-agents; OpenAI AgentKit; Claude Code's Agent/Workflow tools.

Open source

LangGraph (graph-based control flow, the current standard), CrewAI (role-based crews), AutoGen / AG2 (conversational multi-agent), and Qwen-Agent — all point at any OpenAI-compatible endpoint, so Ollama works.

On your M1 Max

Orchestration itself is free — but every sub-agent is another call into the same local model, so "parallel" agents serialize. Keep crews small (2–3 roles).

details & example prompt How it works

Each "agent" is the same LLM called with a different system prompt, tool set and slice of shared state. A graph (LangGraph) or crew definition decides who runs when and what they can see; a supervisor node routes work and judges when it's done. The win isn't parallelism locally — it's focus: a dedicated critic with a narrow prompt catches things a single do-everything prompt misses.

Where it's most useful

Draft→critique→revise writing loops, code-with-reviewer generation, research pipelines where gathering and synthesis want different instructions.

Try this prompt

Set up two agents: a writer that drafts a blog post from my notes, and an editor that critiques it for clarity and cuts 20%. Run one round-trip and show both versions.

Skills — reusable packaged workflows

Frontier

Anthropic Skills: a folder with SKILL.md (YAML frontmatter + instructions) plus optional scripts, loaded progressively. Now an open standard.

Open source

Clone anthropics/skills for the office-document skills; the format is plain Markdown, so any agent framework (Goose, OpenHands, your own loop) can load frontmatter cheaply and expand on demand. Write your own for recurring workflows.

On your M1 Max

Zero cost — it's a prompt-packaging convention, not a runtime. The ~100-token frontmatter trick matters more locally, where context is precious.

details & example prompt How it works

At startup the agent reads only each skill's YAML frontmatter — a name and a "use when…" line, ~100 tokens each. When a task matches, it loads that skill's full SKILL.md instructions into context and runs any bundled scripts in the sandbox. Progressive disclosure is the whole trick: you can have 50 skills installed and pay for one.

Where it's most useful

Anything you do the same way twice — invoice generation, weekly reports in your format, brand-styled documents, personal checklists.

Try this prompt

Use my "weekly-report" skill on the files in ~/reports/inbox and produce the usual docx — same sections and tone as always.

25

Image editing & inpainting

Frontier

GPT-image edits — "remove the person", "change the background", style transfer, targeted region edits.

Open source

Qwen-Image-Edit (GGUF quants exist) for instruction-based edits; ComfyUI inpainting/outpainting workflows with SDXL + ControlNet; Draw Things has inpainting built in with native Metal.

On your M1 Max

Same memory rules as image generation — unload the LLM first. SDXL inpainting is comfortable; instruction-edit models are heavier.

details & example prompt How it works

Inpainting re-runs diffusion only inside a mask you paint, blending the new region with the untouched pixels around it — that's object removal and background swaps. Instruction-edit models (Qwen-Image-Edit) skip the mask: they condition on the original image plus your sentence and regenerate the whole frame with just the requested change.

Where it's most useful

Removing clutter and photobombers, swapping backgrounds for product shots, extending crops (outpainting), quick style passes.

Try this prompt

Remove the power cables and the trash bin from this photo, extend the sky naturally where they were, and keep everything else pixel-identical.

Meeting transcription & diarization

Frontier

Open source

On your M1 Max

Excellent fit — whisper.cpp is Metal-accelerated and an hour of audio transcribes in minutes. Diarization adds a pyannote pass on CPU/MPS.

details & example prompt How it works

You then rename speakers once ("speaker 0 is Priya") and hand the transcript to your LLM for minutes, decisions and action items.

Where it's most useful

Standups and client calls, interviews, lectures, podcast episodes you want notes from — everything you'd otherwise pay Fathom-style tools for.

Try this prompt

Open source

On your M1 Max

details & example prompt How it works

Where it's most useful

Digesting papers and long reports on a commute or run, turning your own notes into revision audio, making dense material stick.

Fine-tuning your own custom model

Frontier

Custom GPTs / Projects personalize with prompts — actual weight customization is API-only fine-tuning.

Open source

mlx-lm LoRA (Apple-native, the easiest path on your hardware), Unsloth (fastest, needs CUDA — cloud/Colab), Axolotl for bigger jobs. Fine-tune tone, format, or domain vocabulary — not new knowledge (use RAG for that).

On your M1 Max

LoRA on a 4-bit 7–14B model is realistic overnight work with mlx-lm; 30B-class tuning wants more memory. Export the fused model back to GGUF for Ollama.

details & example prompt How it works

LoRA freezes the base model and trains small low-rank adapter matrices on a few hundred to a few thousand prompt→response pairs (JSONL). Because only the adapters train, it fits in consumer memory. Afterwards you fuse the adapters into the weights and re-quantize to GGUF so Ollama serves it like any other model. Rule of thumb: fine-tune for style and format, use RAG (card 14) for knowledge.

Where it's most useful

Making drafts sound like you, enforcing a strict output format, domain jargon, high-volume classification where prompting is too slow.

Try this prompt

(the goal) Fine-tune an 8B model on my 500 past support replies so drafted answers match my tone and structure — then A/B it against the base model on 20 held-out tickets.

Guardrails & content moderation

Frontier

OpenAI's moderation endpoint and both vendors' built-in safety layers — filtering inputs/outputs before they reach the model or the user.

Open source

Llama Guard 3 (1B/8B classifier models, run via Ollama), NeMo Guardrails (programmable rails around any LLM), LLM Guard (prompt-injection and PII scanners as a library).

On your M1 Max

Llama Guard 1B adds ~1 GB and one quick pass per message — cheap insurance if you expose your stack to family or the internet.

details & example prompt How it works

A small classifier model sits in front of (and after) your main model: every user message and every reply gets a fast safe/unsafe verdict against a category taxonomy before it's allowed through. NeMo Guardrails goes further with programmable "rails" — topic boundaries, forced tool checks, canned refusals — defined in a config file, not in the prompt where users can talk their way around it.

Where it's most useful

A kid-safe family assistant, anything exposed beyond localhost, and agents with powerful tools where you want a second opinion before actions run.

Try this prompt

(rails config, not chat) Block medical dosage advice and personal-data extraction; route "delete/send/pay" tool calls through a confirmation step; log every refusal to guardrails.log.

Evals, benchmarking & prompt testing

Frontier

OpenAI Evals / platform playgrounds — comparing prompts, models and regressions before shipping.

Open source

promptfoo (declarative prompt/model test matrix, works against Ollama), lm-evaluation-harness (academic benchmarks on your quants), and Open WebUI's built-in model arena for blind A/B votes.

On your M1 Max

Perfect fit — this is how you actually decide between 14B dense and 30B-A3B for your tasks instead of trusting leaderboards.

details & example prompt How it works

You write a YAML file of test cases — prompts, the models to compare, and assertions (contains X, matches regex, valid JSON, or "LLM-as-judge" grading). promptfoo runs the full matrix against your endpoints and renders a pass/fail grid, so a system-prompt tweak that silently breaks case #7 is caught before you ship it to your daily driver.

Where it's most useful

Choosing between models and quants on your real tasks, regression-testing system prompts, verifying a fine-tune actually improved things.

Try this prompt

(promptfoo config) Compare qwen3:14b vs qwen3:30b-a3b on my 20 receipt-extraction cases — assert the output is valid JSON and totals match the expected values.

Model routing & unified gateway

Frontier

GPT-5-style auto-routing — the product silently picks a fast or deep model per request.

Open source

LiteLLM (one OpenAI-compatible proxy in front of Ollama, MLX, llama.cpp and cloud APIs, with fallbacks and budgets), RouteLLM (learned cheap-vs-strong routing), OptiLLM (inference-time optimization proxy).

On your M1 Max

A light Python proxy. The killer local pattern: route easy turns to Qwen3-14B, hard ones to 30B-A3B — or overflow to a paid API only when you choose.

details & example prompt How it works

LiteLLM exposes one OpenAI-compatible endpoint and maps model names to real backends — Ollama, llama.cpp, MLX, or cloud APIs — with per-route fallbacks, retries and budget caps in a YAML config. Every app you own points at the proxy once; behind it you can swap models, add a fallback, or cap costs without touching a single client. RouteLLM adds a learned classifier that sends easy queries to the cheap model automatically.

Where it's most useful

Running fast + deep local models side by side, surviving one backend being down, and controlled overflow to a paid API for the rare too-hard task.

Try this prompt

(gateway config) Route chats under 1k tokens to qwen3:14b and code or long-context requests to qwen3:30b-a3b; on timeout, fall back to the other; hard-cap cloud spend at $5/month.

The glue

The host layer that ties it all together.

MCP-capable chat UIs

LibreChat — native MCP, Agents, Artifacts, Code Interpreter, RAG, multi-user. The most "ChatGPT-like" and the most production-ready. Open WebUI — native MCP plus the mcpo OpenAPI bridge, strong local-model RAG, voice and image-gen. LobeChat and AnythingLLM round out the field.

Model runtimes

Ollama and LM Studio are the easiest (OpenAI-compatible, GGUF). llama.cpp gives the most control and the --jinja tool-calling you'll need. mlx-lm is the fastest on Apple Silicon — but remember it can't call tools, so keep it for plain chat, not agents.

A staged plan

Build it in three calm stages.

Stage 1 · Core stack — do this first

Get a model talking to search, fetch and files.

  1. Install Ollama (or LM Studio) and pull Qwen3-30B-A3B at Q4 — or 14B for maximum headroom. Use Unsloth quants so tool-calling works.
  2. Front it with LibreChat (or Open WebUI) — you instantly get artifacts, RAG, multi-conversation memory and native MCP.
  3. Add three MCP servers: mcp-searxng (search), Playwright MCP (fetch/browse), and the official filesystem server.

Stage 2 · Productivity parity

Make documents, read PDFs, remember things.

  1. Stand up a Docker code-execution path (llm-sandbox or Open Interpreter) with python-docx / openpyxl / python-pptx / WeasyPrint — clone anthropics/skills to reuse Claude's workflows.
  2. Add Docling (or PyMuPDF) for PDFs and a LanceDB / Chroma collection for Projects / knowledge bases.
  3. Add mem0 or the official memory server — move to Letta if recall quality matters more than overhead.

Stage 3 · Advanced & agentic

Research, voice, images, automation.

  1. Run GPT Researcher or Local Deep Research against your Ollama + SearXNG.
  2. Wire whisper.cpp + Kokoro/Piper for voice (via VoiceMode or local-stt-mcp).
  3. Install Draw Things (or ComfyUI + GGUF) with SDXL / Flux schnell — but run it when the LLM is unloaded.
  4. Try browser-use for browser automation; use n8n or launchd for scheduling.

Thresholds that change the plan

  • Tool-calling failing or throwing 500s → switch to Unsloth quants + updated llama.cpp, or community-fixed templates.
  • Model too slow or weak for agent chains → drop to Qwen3-14B, or accept that long-horizon agentic work is where 32 GB most visibly trails the frontier.
  • Need concurrent image-gen + chat, local video, or 256K context → you'll want a second machine with a discrete GPU or 64 GB+. These don't fit 32 GB.

Before you start

Five honest caveats.

  • Memory is the hard limit. Image generation, vision models, voice TTS and your LLM all draw on the same 32 GB. Plan for sequential use. Local video generation is out of scope here.
  • Small models are weaker at long-horizon agentic tasks. Computer use, deep research and multi-step orchestration are exactly where a 14–30B Qwen trails the big models. OSS benchmark scores usually use a frontier brain — expect more supervision locally.
  • Tool-calling template bugs are real and recurring across Qwen3/3.5/3.6 on llama.cpp. Budget time for the Unsloth fixes, and prefer llama.cpp/Ollama/LM Studio over MLX when tools matter.
  • MCP runs inside your trust boundary. Servers can execute code and read data — only add ones you trust, prefer read-only/sandboxed configs. Open WebUI gates MCP to admins for exactly this reason.
  • The ecosystem moves weekly. Model names, registries and benchmarks shift fast — verify current versions before committing. Some frontier feature details here are from vendor docs describing previews rather than fully shipped features.

And one reframing: Claude has no native image generation, and ChatGPT's sandbox has limited internet — so "matching them" sometimes means matching a deliberate limitation, not adding a feature.
